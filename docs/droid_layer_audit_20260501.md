## 1. Leakage の再確認 — **WARN**

### 1.1 clean 化ロジックの確認

3つのスクリプトで clean フィルタの実装を確認:

| スクリプト | フィルタ条件 | 判定 |
|---|---|---|
| `v5_clean_eval.py` (L28-32) | `(cd - rd).days <= 0` | **緩い** — 同日レース後 backfill を含む |
| `full_validation_clean.py` (L93-103) | `cd > rd` は除外、同日は `jst_hour >= 9` で除外 | **適切** — 09:00 JST カットオフ |
| `nar_deep_pop_filter.py` (L117-122) | 同上 (cd > rd 除外 + 同日 jst_hour >= 9 除外) | **適切** |

### 1.2 問題点

**(a) v5_clean_eval.py のフィルタが他2本より緩い**

`v5_clean_eval.py` は `(cd - rd).days <= 0` のみ。同日に当日結果を見てから backfill した行も clean として採用される。v5 レポート (engine_accuracy_audit_v5_FINAL) の根拠スクリプトが `nar_deep_dive.py` / `nar_full_period_analysis_v2.py` であり v5_clean_eval.py とは別だが、v5 の clean 数字がどのスクリプトから導出されたかの追跡性が弱い。

- **影響度**: v5 レポートの 1年データ (13,529 races) は backfill 由来 → backfill 時の created_at は一括実行日 (2026-04-23 前後) のため、ほぼ全行が `created_at >> race_date` → **逆に clean フィルタで大半が除外されるはず**。
- つまり v5 の 1年データは clean フィルタを通していない全量データ。v5 自身がこれを「軽微なリーク」と認識・記載している (v5 レポート §1)。

**(b) backfill データの本質的 leakage**

`backfill_engine_hit_rates_from_pckeiba.py` は「現在のモデルで過去レースを再予想」する (スクリプト冒頭コメント L11-13)。これは:
- エンジンのモデルパラメータが過去データで学習されている場合 → **in-sample 予測**
- エンジンが出走表データのみで予測する (学習なし) 場合 → 問題なし

v5 レポートは「NARは更新影響小」と主張しているが、エンジン4基 (dlogic/ilogic/viewlogic/metalogic) の内部がブラックボックスのため、この主張を検証できない。

**(c) clean 2ヶ月 vs 全1年の使い分け**

- Layer 1 の公式数字 (396.9%) は `nar_deep_pop_filter.py` の clean 2ヶ月 (2026-03~04) から
- v5 の 320.6% は全1年 (backfill 含む) から
- Layer 3 の S級数字は `full_validation_clean.py` の clean 2ヶ月から

2つの異なるデータセットが混在し、配信では "回収率 396.9%" (clean 2ヶ月) を掲示している。

### 1.3 結論

| 項目 | 判定 |
|---|---|
| clean 化ロジック (full_validation, nar_deep_pop_filter) | **PASS** — 09:00 JST カットオフは合理的 |
| clean 化ロジック (v5_clean_eval) | **WARN** — 同日夜のレコードを含む可能性 |
| backfill データの leakage | **WARN** — エンジン内部が不透明なため完全否定不能 |
| clean vs 全量データの混在 | **WARN** — 配信数字の出典を統一すべき |

### 1.4 推奨対応

1. `v5_clean_eval.py` にも 09:00 JST カットオフを追加 (他2本と統一)
2. 配信で使う数字は clean データ由来のみに限定。1年データは参考値として別記
3. エンジン4基の予測ロジックが出走表のみで動くことを明文化 (CLAUDE.md or docs に)

## 2. 統計的妥当性 — **WARN**

### 2.1 Bootstrap CI の実装確認

`full_validation_clean.py` L279-291:
```python
def bootstrap_ci(profits, n_resamples=1000, ci=0.95):
    if not profits or len(profits) < 30:
        return None, None
    means = []
    n = len(profits)
    for _ in range(n_resamples):
        sample = [profits[random.randrange(n)] for _ in range(n)]
        means.append(sum(sample) / n)
    means.sort()
    lo = means[int(n_resamples * (1 - ci) / 2)]
    hi = means[int(n_resamples * (1 + ci) / 2)]
    return (1 + lo / 100) * 100, (1 + hi / 100) * 100
```

- リサンプリング数: 1,000 回 — 標準的には 10,000 回が推奨。1,000 回だと CI 端のばらつきが大きい
- 置換あり: 正しい (standard bootstrap)
- `random.seed(42)` で固定: 再現性あり (PASS)
- パーセンタイル計算: `int(1000 * 0.025) = 25`, `int(1000 * 0.975) = 975` — 正しい

**実装自体は概ね正しいが、n_resamples=1,000 は低め。**

### 2.2 サンプルサイズの評価

| 戦略 | n | 回収率 | CI下限 | CI幅 (上限-下限) | 判定 |
|---|---|---|---|---|---|
| **Layer 1** Z-v3仮説 単勝 | 145 | 396.9% | 225% | 388pt | **WARN** — CI幅が非常に広い |
| **Layer 3** S1 三連複1点 | 372 | 836.9% | 231% | 1521pt | **WARN** — CI幅 1500pt超、外れ値依存 |
| **Layer 3** U2 馬連BOX3 | 1,116 | 325.9% | 213% | 240pt | **PASS** — n 十分 |
| **Layer 3** F5 複勝 | 590 | 130.6% | 118% | 26pt | **PASS** — CI幅狭く安定 |

- Layer 1 (n=145): 中穴単勝なので高配当1発で回収率が大きく変動。**CI下限 225% は信頼に足るが、中心値 397% は楽観的**
- Layer 3 S1 (n=372, CI幅 1521pt): 三連複1点はヒット率2.4%程度。**少数の万馬券が全体を支配**。中央値ベースの回収率はほぼ 0% に近いはず
- Layer 3 F5 (n=590, CI幅 26pt): 複勝は安定しており統計的に最も信頼できる

### 2.3 多重比較問題 (p-hacking)

`full_validation_clean.py` のレポート冒頭に多重比較の注意書きが明記されている (L309-311):
> "Bonferroni補正目安: 100セグメント検定なら有効α=0.05/100=0.0005相当。S/A級採択はあくまで"発見フェーズ"の候補リストであり、独立OOSデータでの再検証を要する。"

**注意書きは正直かつ適切。しかし、実際の運用ではこの警告を無視して S級をそのまま配信に使っている。**

具体的なセグメント検索数:
- `full_validation_clean.py`: 22戦略 × (ALL + RT×2 + VENUE×15 + WD×7) ≒ **550セグメント**
- `nar_deep_pop_filter.py`: 31戦略 × (ALL + VEN×14 + WD×7 + GRP×5) ≒ **800セグメント**
- 合計 **1,350+ セグメント**を同時検定

Bonferroni 補正を適用すると α_eff = 0.05 / 1350 ≒ 0.000037。この厳しい基準で有意になるのは、CI 下限が大幅にプラスの戦略のみ。

**S級 14戦略のうち、この基準を満たすのは F5 複勝 (CI下限 118%) 程度。** 三連複 S1 の CI下限 231% は幅が広すぎて Bonferroni 後も有意かは微妙。

### 2.4 結論

| 項目 | 判定 |
|---|---|
| Bootstrap 実装 | **PASS** (n_resamples を 10,000 に上げることを推奨) |
| Layer 1 n=145 | **WARN** — CI下限 225% は有望だが幅広い |
| Layer 3 S1 三連複 | **WARN** — 外れ値依存、実質的なヒット率が極低 |
| Layer 3 F5 複勝 | **PASS** — 最も安定 |
| Layer 3 U2 馬連 | **PASS** — n=1,116 で CI下限 213% |
| 多重比較 | **WARN** — 注意書きはあるが運用に反映されていない |

### 2.5 推奨対応

1. Bootstrap リサンプリング数を 10,000 に変更
2. S1 三連複は中央値・第1四分位の回収率を別途算出し、外れ値耐性を確認
3. 多重比較を考慮した上で、S級の採択基準をより厳格に (例: CI下限 150% 以上)

## 4. 過剰最適化 — **WARN**

### 4.1 Layer 1 の条件分析

現行 Layer 1 条件: `NAR + 火水木 + 旧強5会場 (門別/船橋/大井/川崎/盛岡) + 6-12頭 + 5-8人気 + 2-3エンジン一致`

条件の数: **6つのフィルタ** を同時適用 → 典型的な過学習シグネチャ。

`pckeiba_long_backtest.py` の結果 (docs/pckeiba_long_backtest_2020_to_2026.md) が決定的:

> **エンジンなし (素朴な統計フィルタのみ) での同条件回収率: 72.2%** (n=66,630, 2020-2026)

つまり:
- フィルタ条件だけでは回収率 72% (大赤字)
- エンジン合議を加えると 397% (clean 2ヶ月) / 320% (1年)

**エンジン合議がエッジの源泉であり、フィルタ条件自体は選別力がない。** これは重要な発見であり、v5 レポートでも正しく認識されている。

### 4.2 条件を緩めた場合の検証

v5 レポートから:

| 条件 | n | 回収率 | 判定 |
|---|---|---|---|
| NAR + 強会場v4 + 6人気 + 2-3一致 + 月-金 (A3) | 255 | 320.6% | 最強 |
| NAR + 南関東4場 + 2-3一致 + 月-金 (A5, 人気不問) | 2,591 | 141.1% | 安定 |
| NAR + 2-3 top1一致 (全NAR, フィルタなし) | — | 104% | ベースライン |

条件を緩めると回収率は急落するが、A5 (141%, n=2,591) はサンプルサイズが十分で安定感がある。

**A3 (320%) vs A5 (141%) のトレードオフ**:
- A3: 高回収率だが n=255、季節変動に脆弱
- A5: 中回収率だが n=2,591、統計的にロバスト

### 4.3 Layer 3 S1 三連複の外れ値検証

S1 三連複 (JRA): 回収率 836.9% (n=372)

三連複1点のヒット率 ≈ 的中数 / n。clean データの JRA レースあたりのヒット率は不明だが、一般的な三連複1点の的中率は 1-3%。n=372 で的中が 5-10 件程度と推定。

**仮に 10 件のヒットのうち最大2件が万馬券 (例: 30,000円超) だった場合、その2件を除外するだけで回収率は半減以下になる可能性がある。**

Mar (331.8%) vs Apr (1310.5%) の乖離が大きい → 4月に高額配当が集中した可能性が高い。

### 4.4 v3 → v5 での学習効果

v5 レポートは正直に記載:
- v3 の 457% → v5 の 150% (同条件で 1年再検証)
- 「短期ハマりだった」と結論

**この経験を踏まえると、v5/v6 の clean 2ヶ月データから導出された 397% も同様の「短期ハマり」リスクがある。** 1年データの 320% がより信頼性の高い推定値。

### 4.5 結論

| 項目 | 判定 |
|---|---|
| Layer 1 フィルタ数 (6条件) | **WARN** — 多いが、エンジン合議がエッジ源泉なのでフィルタ自体の過適合は限定的 |
| Layer 1 A3 vs A5 トレードオフ | **推奨**: A5 を主力、A3 を上乗せとする現行方針は妥当 |
| Layer 3 S1 三連複 外れ値依存 | **WARN** — 高額配当2-3件で全体が変わる |
| v3→v5 での過適合パターン | **WARN** — clean 2ヶ月の 397% は過大評価の可能性あり |

### 4.6 推奨対応

1. 配信の掲示回収率は 1年データ (320%) を採用。2ヶ月データ (397%) は参考値
2. S1 三連複はヒット分布の開示 (中央値、最大配当、Top-3ヒットを除外した回収率) を追加
3. 3ヶ月後 (2026-08 頃) に clean データが十分蓄積されたら再検証を必須化
